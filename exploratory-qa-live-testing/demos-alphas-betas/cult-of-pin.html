Introduction

This case study is part of my Exploratory QA & Live Testing work.

These projects focus on short, real-world testing scenarios such as demos, alpha builds, and unstable environments, where the goal is rapid bug discovery, practical QA judgement, and adapting to evolving systems rather than full structured coverage.

This project is a short exploratory QA pass on the Cult of PiN demo, tested on PC using build v0.11. The goal was to evaluate how clearly the game communicates its ritual systems, how well overlapping systems behave under chaotic gameplay conditions, and whether the run flow remains stable from start to completion.

Testing was charter-driven and risk-based rather than exhaustive. The goal was not full regression or full content coverage, but to focus on high-risk areas in a system-heavy roguelike pinball game where multiple mechanics can overlap quickly.

I did not use Jira for this project. Bugs and findings were recorded directly in a structured QA workbook using session notes, a bug log, risk matrix, one-line summary, and STAR summary.

⭐ STAR Summary - Cult of PiN Demo Exploratory QA

Situation: Short, self-directed exploratory QA pass on the Cult of PiN demo, a system-heavy roguelike pinball game where rituals, multipliers, multiball, and boss mechanics can overlap quickly during play.

Task: Validate high-risk areas in the demo build, focusing on clarity, ritual behaviour, timing and state transitions, system stacking, and progression flow.

Action: Created a charter-driven QA workbook and executed 5 focused sessions. Testing prioritised real gameplay conditions over exhaustive coverage, with evidence captured through screenshots and short gameplay clips. Bugs were recorded in a structured Bug Log rather than Jira.

Result: Logged 1 low-severity visual consistency issue. No timing, system interaction, or progression defects were observed. The main project finding was that the game is mechanically stable under pressure, but player-facing clarity weakens when multiple systems overlap.

This was logged as a visual feedback issue rather than a mechanics defect. Rituals establish clear colour mapping between UI and in-game effects, but Spare Change breaks that mapping by appearing purple in the HUD while using red and yellow gameplay effects.

📈 Results

  • Completed 5 focused exploratory QA sessions on the Cult of PiN demo.
  • Logged 1 low-severity visual consistency bug with 100% repro.
  • Confirmed no timing, system interaction, or progression flow defects during tested conditions.
  • Identified clarity and feedback as the main player-facing weakness.
  • Confirmed that overlapping systems can function correctly while still reducing player understanding.

🧠 What I learned

  • Stable systems can still create player-facing confusion. The game’s mechanics behaved consistently, but feedback did not always scale well when multiple systems overlapped.
  • Not every finding needs to become a bug. Some observations were better recorded as session findings rather than forced into defect reports.
  • Colour coding matters in fast gameplay. Consistent visual language helps players recognise effects quickly, especially during chaotic moments.
  • Exploratory QA still needs structure. Charters helped keep testing focused without turning a short demo pass into fake exhaustive coverage.
  • Knowing when to stop testing is part of QA judgment. After validating clarity, timing, stacking, and progression, further testing would have added more noise than value.

🔚 Conclusion

This short exploratory QA pass on Cult of PiN found that the demo is mechanically stable across the tested timing, stacking, and progression areas. No timing defects, interaction conflicts, or progression flow failures were observed.

The main weakness was player-facing clarity. Rituals and overlapping systems often function correctly, but the game does not always make cause-and-effect clear during chaotic play. The logged Spare Change colour mismatch reinforced this wider theme by showing how inconsistent visual mapping can reduce quick recognition during gameplay.

Overall, this project demonstrates risk-based exploratory testing, clear separation between bugs and findings, and practical evidence capture for a short live demo build.
