How I Structure Game QA Evidence

From test session to recruiter-ready Jira bug report for game teams.

Author: Kelina Cowell (self-directed game QA portfolio work) • LinkedIn

This article explains the evidence workflow I use across my game QA portfolio projects: recording full gameplay sessions, extracting focused clips, tracking findings in Google Sheets, and creating linked Jira bug reports with clear traceability from game test session to final report.

TL;DR

  • Core workflow: Test → Capture → Clip → Label → Store → Report.
  • Tools used: OBS Studio, Microsoft Clipchamp, Google Sheets, Jira, YouTube (unlisted), and external SSD backup.
  • Main point: good game QA evidence is not just finding bugs, but making them easy to reproduce, understand, and act on.
  • Traceability: my workbook, evidence clips, and Jira tickets are linked together as one pipeline.
  • Examples shown: functional, exploratory, and regression game testing from Battletoads, Rebel Racing, Sworn, Shadow Point, and The Chef’s Shift.

What this article is for

This page exists to show how I work in practice as an aspiring game QA tester. Rather than only showing finished bug reports, I wanted to show the full chain behind them: how gameplay sessions are recorded, how evidence is trimmed, how game test documentation is structured, and how issues are reported clearly in Jira with supporting clips and notes.

Why Game QA Evidence Matters

Finding a bug isn’t enough.

If a game developer can’t see it, reproduce it, and understand it quickly, the bug slows the team down instead of helping it.

That’s why I focus on structured game QA evidence capture — turning raw gameplay into clear, reproducible, developer-ready bug reports for game teams.

My Game QA Evidence Workflow

Every issue I log follows the same pipeline:

Core workflow

Test → Capture → Clip → Label → Store → Report

This ensures nothing is missed, and every game bug is backed by clear, usable evidence.

1. Test (Structured or Exploratory)

I begin with either:

  • Structured game testing (test matrix, defined scenarios)
  • Exploratory game testing (charter-based sessions)

Each session is scoped and time-boxed to keep testing focused and intentional.

2. Capture (Full Session Recording)

All game testing sessions are recorded using OBS Studio.

This allows me to:

  • Capture issues in real time
  • Retain full context
  • Avoid relying on memory

When a bug occurs, the evidence already exists.

3. Clip Extraction (Turning Raw Gameplay into Evidence)

Raw recordings are not useful on their own.

Developers don’t want to watch 5–10 minutes of gameplay footage to find a bug.

I use Microsoft Clipchamp to extract short, focused clips that show:

  • The trigger
  • The failure
  • The result

Example: Evidence Refinement

Before → After

Full gameplay session reduced to a short, reproducible clip. This ensures game developers see only what matters, without wasting time searching through raw footage.

4. Naming & Labelling (Making Evidence Traceable)

All clips follow a consistent naming convention:

[PROJECT]_[BUG-ID]_[DESCRIPTION]

Example:

CHEF-1_PaymentFails_MultiCustomer_Run3.mp4

This allows:

  • Instant identification
  • Direct mapping to bug reports
  • Clean organisation across game QA projects

5. Storage & Documentation

I separate game QA data across local storage, cloud access, and structured documentation.

Raw Sessions (Local)

  • Full gameplay recordings stored on my PC
  • Used to revisit context when needed

Edited Clips (YouTube – Unlisted)

  • Uploaded to a dedicated game QA evidence channel
  • Organised by project
  • Linked directly in bug reports

Why YouTube:

  • Avoids local storage overload
  • Fast sharing via link
  • Easy access for reviewers
  • Simulates controlled sharing (similar to NDA workflows)

Documentation (Google Sheets)

Each project uses a single Google Sheets workbook as the game QA hub.

Core tabs:

  • README — scope, environment, constraints
  • STAR Summary (Bugs) — key findings

Additional tabs vary by project:

  • Charter Matrix
  • Session Log
  • Bug Log
  • Device Matrix
  • Daily Smoke

Example: Game QA Workbook Structure

6. Traceability (Connecting Everything)

All evidence is linked directly inside the workbook:

  • Bug IDs → video clips
  • Test sessions → notes and outcomes
  • Findings → summarised in STAR format

This creates a clear chain:

Traceability chain

Test → Observation → Evidence → Report

7. Bug Reporting (Jira)

I document all issues in Jira using a structured, consistent format.

Each bug includes:

  • Title
  • System / Area
  • Environment
  • Build
  • Reproduction Steps
  • Expected Result
  • Actual Result
  • Severity / Priority
  • Repro Rate
  • Evidence (linked video/screenshots)

Example: Jira Game Bug Report

All Jira tickets are directly linked to evidence clips and tracked within my QA workbook for full traceability.

8. Game QA Pipeline: Sheets ↔ Jira Integration

My workflow connects documentation and reporting into a single system rather than treating them as separate tools.

How it connects

Google Sheets (Tracking & Context)

  • Game test runs and session logs
  • Bug discovery and notes
  • Evidence links
  • Internal bug ID tracking
⬇

Jira (Formal Reporting)

  • Structured game bug ticket
  • Developer-facing documentation
  • Severity and priority assignment

Field Alignment (Example)

Google Sheets Jira
Bug ID Issue Key
Notes / Observation Description
Test Steps / Actions Reproduction Steps
Expected / Actual (logged) Expected vs Actual
Severity / Priority Severity / Priority
Evidence Link (YouTube) Attachments / Links

This ensures:

  • No duplication of effort
  • Consistent data across tools
  • Clear traceability from game test session → evidence → bug report

Backup Strategy (Critical)

All game QA data is backed up to an external SSD, including:

  • Raw recordings
  • Supporting files
  • Documentation

Why This Approach Works

This workflow:

  • Mirrors real game QA processes
  • Produces clear, reproducible game bugs
  • Reduces developer friction
  • Demonstrates structured thinking

Most importantly:

It shows not just that I can find issues — but that I can communicate them effectively in a game production environment.

Final Thought

Good game QA isn’t about finding the most bugs.

It’s about making sure the right bugs get fixed quickly.

And that comes down to how well you capture and present evidence.
