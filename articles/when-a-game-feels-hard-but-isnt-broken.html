When a game feels hard… but isn’t broken

An applied QA article about accessibility, cognition, and player-facing friction under pressure.

Author: Kelina Cowell (self-directed QA portfolio work) • LinkedIn

This article is based on my short early build QA pass on The Chef’s Shift (PC, v0.1.2a). It exists to show how player struggle is not always a skill issue, and how UI clarity, multitasking load, and prompt visibility can make a game feel harder even when the underlying system still works.

TL;DR

  • Point: not every player struggle is a system bug or a skill issue.
  • Example context: The Chef’s Shift on PC, tested in a 2-day early build pass across 8 logged sessions.
  • Main example: CHEF-2 showed that poor prompt visibility made an interaction feel harder without the system actually failing.
  • QA value: accessibility-aware testing helps separate true system failure from cognitive overload and poor communication.
  • Takeaway: difficulty is not automatically good design if the player is fighting clarity instead of challenge.

Scope included core loop stability, input handling, serving flow, recovery, and accessibility-related pressure points. The goal was not a full accessibility audit, but a realistic QA pass that also examined how readability, typing speed, and multitasking pressure affected player-facing friction.

When a game feels hard… but isn’t broken

One of the easiest mistakes to make in QA is assuming that if a player struggles, the problem is skill.

Sometimes that is true.

But sometimes the game is not hard. It is just unclear.

Core idea

Difficulty is not the same as good design. If the player fails because the game communicates poorly, that is not challenge. It is friction.

Why this matters in early build QA

In early builds, it is easy to misread player behaviour.

You see hesitation, mistakes, or slow performance and assume:

  • The player needs practice
  • The mechanic is intentionally demanding
  • The difficulty curve is working

But in a typing-based game like The Chef’s Shift, performance is heavily influenced by:

  • How clearly prompts are presented
  • How quickly information can be recognised
  • How much the player has to track at once

If those break down, the player looks like they are struggling, even when the system itself still works.

That matters in QA because it changes the diagnosis. A rough player moment is not always proof of broken logic. Sometimes it is proof that the game is asking the player to do too much with too little support.

Example: CHEF-2 and prompt visibility

In my early build pass, the clearest example of “this feels harder than it should” was CHEF-2.

The oven interaction prompt was technically present. The system worked. The interaction could be completed.

But under pressure, the prompt was difficult to see against the background during the pizza step.

The result was not a system failure. It was something more subtle:

  • Delayed recognition
  • Missed interactions
  • Hesitation before acting
  • Breaks in serving flow

From the outside, this can look like player error.

In practice, it is a communication problem.

Key distinction

The system allowed success. The UI made success harder to achieve quickly.

Typing under pressure is not just input

At low complexity, performance in The Chef’s Shift is mostly about typing accuracy.

You see the prompt, you type the word, you move on.

But as complexity increases, that changes.

The player is no longer just typing. They are:

  • Tracking multiple customers
  • Managing order flow
  • Watching for the next interaction prompt
  • Trying not to fall behind while still typing correctly

Typing becomes one part of a larger cognitive load rather than a standalone input task.

What changed in this project

In my testing, performance shifted from input accuracy at low complexity to cognitive load and task management under higher pressure.

Multitasking is where clarity matters most

This is where design either supports the player or works against them.

When multiple tasks are active, the player relies on fast recognition, not careful reading.

They need to:

  • Spot the right prompt instantly
  • Understand what action is required without second-guessing
  • Act before another demand appears

If the UI slows that process down, even slightly, the effect compounds.

The player hesitates. They lose tempo. They start making mistakes. The game now feels harder.

But the difficulty is not necessarily coming from the mechanic itself. It is coming from the cost of understanding what to do.

Accessibility angle

This is where readability, prompt contrast, recognition speed, and multitasking load stop being “nice extras” and become part of whether the experience feels fair.

Difficulty vs communication

This was the core distinction that shaped how I interpreted the results of this project.

  • Real difficulty: the player understands the task, but execution is demanding
  • Poor communication: the player struggles to understand the task quickly enough to act

Both can produce failure.

Only one reflects intended challenge.

When a player fails, the real question is not “was that hard?” It is “did they have a fair chance to understand what to do?”

This is why I do not treat every rough player moment the same way. A hard mechanic can be valid. A badly communicated one is a different problem.

In a pressure-driven loop, poor communication can imitate difficulty so well that teams risk mistaking friction for design strength.

What this means for QA

This is where accessibility-aware testing adds value without becoming preachy or turning every issue into a moral lecture.

The point is not to label everything as an accessibility defect.

The point is to separate:

  • Systems that are breaking
  • Players who are being overloaded
  • Interfaces that are not communicating clearly enough

In this project, that distinction mattered.

CHEF-1 was a genuine system failure. It blocked progression through inconsistent payment behaviour under multi-customer conditions.

CHEF-2 was different. The system still functioned, but prompt visibility created friction that slowed recognition and action under pressure.

Treating both as generic bugs would miss the point. One broke the loop. The other made the loop feel harsher than it needed to.

Practical QA value

Being able to explain why something feels hard is often more useful than simply proving that it happens.

Takeaway

Not every difficult moment in a game is good difficulty.

Sometimes the system works, but the player is still being set up to fail.

In a typing-based, pressure-driven loop, small issues like prompt visibility, recognition speed, and multitasking load can shape the experience as much as the core mechanic itself.

That is why good QA should not only ask:

  • Does this work?

It should also ask:

  • Can the player understand this fast enough to succeed?

Because if the answer is no, the game may feel hard without actually being broken.

And that is not meaningful challenge.

That is friction.

QA and accessibility FAQ

Is every frustrating moment an accessibility issue?

No. Some moments are meant to be demanding. The point is to tell the difference between intended challenge and avoidable player-facing friction.

Can a system be technically functional but still create a real issue?

Yes. CHEF-2 is exactly that kind of case. The interaction worked, but the prompt was hard to notice quickly under pressure, which affected performance.

Why does cognitive load matter in QA?

Because players do not interact with one mechanic in a vacuum. They process timing, prompts, targets, and consequences at the same time. That affects how difficulty is experienced.

Is this the same as doing a full accessibility audit?

No. This project was an early build QA pass with an accessibility-informed lens, not a full audit. The goal was to improve interpretation of player-facing friction, not claim complete accessibility coverage.

Why not just say “the player needs more practice”?

Because that can hide real design problems. If the player is failing because the game communicates poorly, practice is not the real fix.

What did this article help show about your QA approach?

That I do not just log what breaks. I also look at how readability, multitasking, and cognitive demand affect the player experience, and I can explain the difference clearly.
