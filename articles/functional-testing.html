Functional testing: the boring basics that catch real bugs

Kelina Cowell • 17 Dec 2025 • ~4 min read

This article is based on a one-week solo functional pass on Battletoads (PC, Game Pass). If the testing sounds basic, that is intentional. In this project, the highest-impact defects lived in the fundamentals.

What functional testing was in this project

In this Battletoads pass, functional testing meant validating whether the game behaved as expected end-to-end under realistic player conditions, using both controller and keyboard on PC. Core flows were validated first, followed by focused checks where risk appeared.

Scope was deliberately constrained to expose defects early: Battletoads (PC, Game Pass), Build 1.1F.42718, tested on Windows 11 at 1920×1080 @144Hz with VRR and V-Sync enabled.

The main output

Clear pass or fail outcomes, backed by short evidence clips and reproducible steps. No interpretation, no assumptions, no opinion-based reporting.

The two flows I tested first

  1. Start → first control
    From Title into Level 1 (L1-A1), the player must reliably gain control and progress through the first arena. In Battletoads, early progression includes an arena exit with no explicit UI cue, making this a high-risk flow to validate first.
  2. Pause / Resume
    Pause interacts with input routing, UI focus, audio muting, HUD state, and co-op joins. In this project, Pause testing surfaced the majority of functional defects.

Why input ownership became its own mini-suite

Battletoads on PC supports both controller and keyboard. That makes mixed input the default state, not an edge case.

  • Pause must respond consistently to the active input method.
  • Navigation focus must not silently switch or become trapped.
  • Confirm and back actions must not misroute to unrelated UI.
  • Disconnecting or reconnecting input devices must not break control.

During this pass, keyboard ↔ controller hand-off produced multiple high-severity defects involving Pause and the Join In overlay, all reproducible within the first level.

Observed pattern

Controller connected + keyboard input + Pause or Join In = focus and routing failures. These issues were consistent, repeatable, and isolated without late-game complexity.

Evidence: what I captured and why

Short video clips (10–30 seconds) were the primary evidence format. These issues depended on timing, input, and focus state, which screenshots could not capture reliably.

  • Video: shows input, timing, and incorrect outcome together.
  • Screenshot: supports static UI state or control bindings.
  • Environment: build, platform, input devices, and display configuration.

If the evidence could not immediately show what was pressed, what happened, and what should have happened, it was replaced.

How this functional pass was timeboxed

  • Early runs: Smoke testing and baseline validation (start flow, Pause open/close).
  • Mid-pass: Expanded testing around Pause, input ownership, HUD, and audio.
  • Late runs: Repetition, confirmation, and isolation of mixed-input defects.
  • Final day: Outcome summary and documentation.
