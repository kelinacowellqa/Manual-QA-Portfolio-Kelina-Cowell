Functional testing: the boring basics that catch real bugs

Kelina Cowell • 17 Dec 2025 • ~4 min read

This is the exact style of functional testing I run in a timeboxed solo pass. If it sounds basic, that’s because it is. Basic is where the breakage lives.

What functional testing is (in practice)

For me, functional testing is simple: does the product do what it claims to do, end-to-end, without excuses or interpretation? I validate core flows, confirm expected behaviour, and write issues so a developer can reproduce them without guessing.

The mistake is treating functional testing as “easy” and therefore less valuable. It is the foundation. If the foundation is cracked, everything built on top of it fails in more complicated ways.

Project context (for reference)

This article is backed by a one-week solo functional pass on Battletoads (PC, Game Pass) using Windows 11 and mixed input (controller plus keyboard). Build tested: 1.1F.42718.

The main output

Clear outcomes: pass or fail, with evidence and reproducible steps. Not vibes. Not opinions.

The two flows I test first

  1. Start → first control
    The first minute determines whether the product feels broken. If “New Game” does not reliably get you playing, nothing else matters. In Battletoads, I validated this from Title into Level 1 (L1-A1) and through the first arena transition before expanding scope.
  2. Pause / Resume
    Pause stresses state, focus, input context, UI navigation and overlays. If Pause is unstable, you get a stream of bugs that look random but are not. In Battletoads (PC), Pause testing surfaced high-impact failures early in Level 1, including keyboard parity and input routing issues when a controller was connected.

Why input ownership deserves its own mini-suite

Mixed input is a feature, not an edge case. When a controller is connected and a keyboard is used, behaviour must remain predictable.

  • Pause must open consistently.
  • Navigation must respect the active input method.
  • Confirm and back must not silently stop responding.
  • Hand-off must not route to the wrong UI or disable input.

I treat keyboard ↔ controller hand-off as a dedicated test area because it produces high-impact, easily reproducible defects. In Battletoads (PC), keyboard actions from Pause could route into Join In, and in one case this left the controller unresponsive until the overlay was closed.

Common pattern

Controller connected + keyboard input + menu open = focus bugs. Easy to reproduce. Easy to prove. Easy to fix once isolated.

Evidence: what I capture and why

I favour short video clips (10–30 seconds) and only use screenshots when they add clarity. The goal is to make the defect obvious without forcing someone to scrub a long recording.

  • Video: shows timing, input, and incorrect outcome together.
  • Screenshot: supports UI state, text, or configuration.
  • Environment: platform, build/version, input device, display mode.

If the evidence cannot answer what was pressed, what happened, and what should have happened, it is not evidence.

In this Battletoads pass, short clips were essential because input and focus state changed quickly, and screenshots could not prove routing issues or temporary loss of control.

How I timebox a functional pass

  • Day 1: Smoke testing and baseline flow validation.
  • Days 2–4: Execute runs, expand where risk appears, log defects immediately.
  • Days 5–6: Retest, tighten repro steps, confirm consistency.
  • Day 7: Summarise outcomes and document learnings.

Practical note: I start each session with a short baseline loop (load, gain control, Pause, resume, exit) before running deeper checks. It catches obvious breakage early and prevents wasted time.

What this looked like in a real project

In the Battletoads functional pass (27 Oct to 1 Nov 2025), the most impactful defects were found in foundational flows within the first level. They did not require late-game content or unusual edge conditions to surface. That is why I front-load functional testing and treat input ownership as a first-class area on PC.
