Functional testing: the boring basics that catch real bugs

Kelina Cowell • 17 Dec 2025 • ~4 min read

This is the exact style of “functional testing” I run in a timeboxed solo pass. If it sounds basic, that’s because it is. Basic is where the breakage lives.

What functional testing is (in practice)

For me, functional testing is: “Does the product do what it says it does, end-to-end, without me having to invent excuses for it?” I’m validating core flows, confirming expected behaviour, and writing issues in a way that a developer can reproduce without guessing.

The trap is thinking functional testing is “easy” and therefore “less valuable”. It’s not. It’s the foundation. If the foundation is cracked, everything built on top of it is just more complicated failure.

The main output

A clean set of outcomes you can point at: Pass/Fail, plus evidence, plus clear steps. Not vibes.

The two flows I test first

  1. Start → first control
    The first minute is where people decide whether the product “feels broken”. If “New Game” doesn’t reliably get you playing, nothing else matters.
  2. Pause/Resume
    Pause is a stress point for state, focus, input context, overlays, audio, and UI navigation. If Pause is flaky, you get a constant stream of “random” bugs that are actually state issues.

Why input ownership deserves its own mini-suite

Mixed input is not a “nice-to-have”. It’s a feature. If a controller is connected and a user touches the keyboard, the product should behave predictably. That means:

  • Pause should open consistently.
  • Navigation should respect the active input method.
  • Confirm/back should not silently stop working.
  • Hand-off should not route you into the wrong overlay or disable the other input.

This is why I treat keyboard ↔ controller hand-off like a dedicated test area. It produces high-impact defects quickly, and they are usually very reproducible when you isolate the conditions.

Example pattern

“Controller connected” + “keyboard key pressed” + “menu open” = focus bugs.
Easy to reproduce. Easy to prove. Easy to fix once you can see it.

Evidence: what I capture and why

I prefer short clips (10–30 seconds) and a screenshot only if it adds clarity. The goal is to make the defect obvious without forcing someone to watch a 6-minute recording.

  • Video: shows input, timing, and the wrong outcome in one go.
  • Screenshot: supports UI text, state, or configuration.
  • Environment details: platform, build/version, input device, display mode.

If your evidence can’t answer “what did you press, what happened, what should have happened?”, it’s not evidence. It’s decoration.

How I timebox a functional pass

In a one-week solo project, I keep it simple:

  • Day 1: smoke + baseline flows + build a compact suite.
  • Days 2–4: execute runs, expand where risk appears, log defects immediately with evidence.
  • Days 5–6: retest, tighten repro, confirm consistency, clean up wording.
  • Day 7: summarise outcomes (metrics, what I learned, next steps).
