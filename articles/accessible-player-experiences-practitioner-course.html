What I learned from the APX (Accessible Player Experiences) Practitioner course

Reflections on accessibility through a QA lens.

Author: Kelina Cowell • LinkedIn

On this page
  • The core insight
  • Key takeaways
  • How this changes my QA approach
  • What comes next

I recently completed the APX Practitioner course by AbleGamers, and it fundamentally changed how I think about accessibility in games.

Accessibility isn’t about making games easier. It’s about making experiences reachable.

The Core Insight: The Player–Game Loop

APX frames games as a feedback loop between player and game.

  • The player inputs an action.
  • The game responds with feedback.
  • The player interprets that feedback and acts again.

If a player cannot input reliably or cannot receive feedback clearly, they are locked out of the loop entirely.

Accessibility work is about restoring that loop. Not removing challenge. Not lowering difficulty. Removing unnecessary barriers that prevent participation.

That reframing alone changed how I evaluate mechanics during testing.

Key Takeaways That Shifted My Thinking

1. Access Comes Before Challenge

Before difficulty matters, access must exist.

  • Cannot remap controls
  • Cannot perceive essential feedback
  • Cannot sustain required physical input

Then difficulty settings are irrelevant.

As QA, this pushes me to test for input reliability under strain, clarity of feedback channels, and whether core mechanics are physically sustainable.

2. Patterns > Checklists

APX teaches design patterns, not rigid compliance rules.

  • Same Controls, But Different
  • Second Channel
  • Do More With Less
  • Slow It Down

Instead of asking “Does the game have accessibility features?” I now ask: “Where is this pattern being applied, and where is it breaking down?”

That shifts bug reporting from reactive to analytical.

3. One Barrier Can Affect Many Players

Accessibility is not diagnosis-based.

  • Permanent disability
  • Temporary injury
  • Ageing players
  • Situational limitations

Accessibility is not niche. It is universal design pressure.

4. Accessibility Is an Experience Problem

The real question is:

Can players still experience connection, challenge, mastery, escapism, and creativity?

That shifts accessibility from a settings menu conversation to a player experience conversation.

How This Changes My QA Approach

  • A framework for identifying genuine player blockers
  • Stronger vocabulary for accessibility-focused bug reports
  • Pattern-based analysis instead of surface-level checks
  • Earlier intervention during development, not just polish-stage validation

Accessibility done well does not dilute a game’s vision. It strengthens it.

This framework will influence every future test plan I write.

What Comes Next

The certificate is not the achievement. Application is.

Next step: integrating APX patterns into live test cases, portfolio projects, and real development workflows.

This isn’t a finished chapter. It’s a new baseline.
