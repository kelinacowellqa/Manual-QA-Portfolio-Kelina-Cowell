Introduction

One-week, change-driven regression pass on Sworn (PC Game Pass, Windows, build 1.01.0.1039) focused on golden-path stability (launch ‚Üí play ‚Üí quit ‚Üí relaunch), save/load integrity, core menus, audio sanity, input handover, and areas plausibly touched by the latest public Steam patch notes. This was a solo, portfolio-only project using SteamDB patch notes as an external oracle, without claiming parity with the Steam build.

  • Scope: change-driven, risk-based regression against the current PC Game Pass build, covering golden-path flow (launch ‚Üí main menu ‚Üí run ‚Üí quit ‚Üí relaunch), autosave Continue behaviour, settings navigation and persistence, death ‚Üí Stats ‚Üí hub flow, audio continuity and settings, Enhancements/Codex readability and persistence, power-up next-room behaviour, pause/Alt+Tab sanity, and controller ‚Üî mouse/keyboard handover with one hot plug. Onboarding accessibility issues first found on Day 0 (SWOR-1 to SWOR-4) are still logged as full bugs, but in the Regression Matrix they are treated as a single known A11y cluster because there was no newer build to re-test.
  • Approach: one-week solo pass built around a Regression Matrix and time-boxed Session Log, with each check mapped to a specific patch-note item or baseline risk area. SteamDB patch notes for SWORN 1.0 Patch #3 (v1.0.3.1111, 13 Nov 2025) were used as an upstream oracle, but only as guidance. Each session captured clear anchors, expected outcomes, and results (Pass, Fail, Not applicable, Known issue), with focused rechecks where needed (for example reconfirming the death-flow defect in SWOR-6).
  • Outcome: Regression Matrix result: 8 passes, 1 fail, 1 not applicable, and 1 known A11y cluster. Across the whole project I tracked six defects in the bug log: four onboarding accessibility prompt issues (SWOR-1‚ÄìSWOR-4), one post-death flow defect where the Defeat overlay blocks Stats and Continue starts a new run (SWOR-6), and one Blessing choice readability issue (SWOR-15). Together they describe the current A11y and flow risks on this build.
  • Evidence: Google Sheets workbook (README, 1-liner summary, Regression Matrix, Session Log, Bug Log, STAR Summary, Glossary) plus OBS/Xbox Game Bar recordings uploaded to YouTube, grouped by area (baseline flow, settings, save/load, death flow, audio, Enhancements, power-ups, Alt+Tab, input handover, economy). All checks and bugs are cross-linked from the Matrix and Session Log so reviewers can jump straight from the portfolio page into the exact clip or log entry.

‚≠ê STAR SUMMARY ‚Äì Sworn Regression QA (PC Game Pass)

Situation: One-week, change-driven regression pass on Sworn (PC Game Pass, Windows, build 1.01.0.1039). The SteamDB patch notes for v1.0.3.1111 were used as an external oracle to guide what areas might reasonably need rechecking, without claiming parity between platforms.

Task: Validate that key systems on the PC Game Pass build behaved correctly after upstream changes ‚Äì golden-path stability, save/load, settings persistence, audio, death-flow logic, Enhancements/Codex behaviour, power-ups, Alt+Tab, and input handover ‚Äì and keep previously identified onboarding accessibility issues visible as a known risk cluster on this build.

Action: Built a Regression Matrix mapped to patch-note items and baseline risks, then executed targeted sessions across the week. Each check was time-boxed with clear anchors and expected outcomes, and marked as Pass, Fail, Not applicable or Known issue. All evidence was captured using OBS/Xbox Game Bar and logged into a structured Google Sheets workbook with a Bug Log (six defects: SWOR-1‚ÄìSWOR-4, SWOR-6, SWOR-15), STAR Summary and fully linked Session Log.

Result: The Regression Matrix outcome was 8 passes, 1 fail, 1 not applicable and 1 known A11y cluster. Most areas passed as expected, including launch‚Üíplay‚Üíquit‚Üírelaunch, autosave Continue, settings navigation/persistence, audio continuity, Enhancements/Codex behaviour, power-up behaviour, Alt+Tab, and controller‚ÜîKBM handover with one hot plug. One regression-relevant defect was confirmed: SWOR-6 (Defeat overlay remains foregrounded, blocking Stats, and Continue starts a new run). A small visual-readability issue in the Blessing choice screen (SWOR-15) was also documented. Onboarding accessibility prompt issues (SWOR-1 to SWOR-4) are tracked in the bug log as a single A11y cluster on this build, but were not re-run in new sessions because no newer build was available.

ü§ù Networking & Applied Insight

During this project I did not guess the scope in isolation, I treated it like a mini live-ops assignment and shaped it around advice from senior QA leads.

Nathan Glatus (ex Senior QA / Game Integrity Analyst, Fortnite, Epic Games) helped me set the initial scope. His advice was to treat Rebel Racing as what it is: a live mobile F2P racer, not a lab toy for every possible edge case. That translated into a small set of focused charters rather than a giant ‚Äútest everything‚Äù list: daily golden-path smoke on a single physical device, input and handling tiers, collisions / exits and race flow, and UI scaling and key live surfaces (store, events, hub). He also pushed me to keep runs to realistic 45‚Äì60 minute sessions with clear exit criteria, and to write tighter bug reports with strong oracles, clear summaries and bundled evidence (video, device/build metadata, repro steps, repro rate) instead of vague ‚Äúit feels off‚Äù notes. His framing around realistic coverage on an approved device list is why this case study is scoped to one main phone but documented in a way that could scale to a real QA team.

Radu Posoi (Founder, AlkoTech Labs, ex Ubisoft QA Lead) then helped me iterate the scope so it matched how mobile QA is actually run day to day. His feedback led me to define clear performance anchors (hub, pre-race, race start, mid-race, results) instead of vague ‚Äúseems fine‚Äù checks; treat interruptions as a first-class surface covering lock screen, app switching, app kill and recovery, and notification shade; and turn battery, heat and multi-touch stress into dedicated charters rather than random one-off experiments. Because Rebel Racing blocks standard Android Studio emulators on the Play Store, he also recommended using Bluestacks as a visual-only oracle for odd aspect ratios and layout stretch while keeping all real testing and bug reproduction on my physical Moto g54. That combination turned my original ‚Äúnice to have‚Äù ideas into a concrete device and network approach that looks like a small slice of a real mobile QA lab rather than a student project.

Their insight directly shaped the final list of charters, including the extra lock/unlock interruption runs that revealed the audio issue on the Results/Rewards screen (RR-37). It also shaped how I recorded device and network context, how I wrote and prioritised bug reports, and the STAR summary for this case study, so the project reads more like a realistic live mobile QA engagement than a purely academic exercise.

  • General stability: no stutters, hitches or progression blockers occurred during any run. All sessions cleanly returned to main menu and relaunched without issue.

üèÅ Result and takeaway

Result: The PC Game Pass build of Sworn remained stable across all golden-path and targeted regression checks. Six defects were logged in total: four onboarding accessibility prompt issues (SWOR-1‚ÄìSWOR-4), one medium-severity death-flow defect (SWOR-6), and one medium-severity Blessing choice readability issue (SWOR-15). All reproduced consistently and are fully documented with evidence.

Takeaway: Sworn‚Äôs core systems are generally robust, but UI readability and onboarding prompt visibility need attention, especially for accessibility. The death-flow defect (SWOR-6) is the highest-impact issue and should be prioritised, while the A11y cluster highlights the need for clearer tutorial cueing in early game moments.

üß† What I learned

  • Controlled scope is the only way regression stays meaningful. Mapping each check to a patch-note item or a clear risk area prevented the week from turning into an unfocused re-test of the whole game and kept every session intentional.
  • Short, purposeful evidence beats long recordings. Tight clips of the golden path, menu flows and the death-flow defect (SWOR-6) made review fast and kept the workbook clean. Anything longer would have buried the signal.
  • The Regression Matrix should do the heavy lifting. Being explicit about expected vs actual outcomes made the STAR summary almost automatic and turned the project into a coherent story instead of a pile of notes.
  • Readability issues aren‚Äôt cosmetic. The Blessing choice prompts (SWOR-15) highlighted how small, low-contrast UI genuinely slows decision-making and can exclude players who rely on clear visual cues.
  • Audio continuity matters to perceived polish. Missing or inconsistent settings ‚Äî like the Dialogue Volume slider referenced upstream but absent here ‚Äî still affect player trust even if the game ‚Äúworks‚Äù.
  • Good notes pay you back immediately. Clear Session Log entries meant I could pick up the project the next day without relearning anything, which kept the week efficient and consistent.
  • Dyslexia surfaces UX problems early. Dense text and cramped layouts in Enhancements and the Codex stood out instantly, which helped justify why these readability issues matter from an accessibility perspective.
  • Admin should help, not hinder. Keeping IDs simple, avoiding duplication and letting the workbook be the single source of truth made the project easy to maintain and easy to audit.
