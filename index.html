1: Functional Testing - Battletoads (PC)

  • Goal: Run a short functional test pass on Battletoads (PC Game Pass) to validate the core loop and surface repeatable bugs, with extra attention on keyboard ‚áÑ controller hand-off.
  • Scope: Core gameplay flow, UI/menus, local co-op, input parity (keyboard + controller), audio cues and basic performance sanity.
  • Focus areas: Pause / Join-In / Resume flows, controller ownership and hand-off, local co-op edge cases, and menu navigation / on-screen prompts.
  • Tools: Jira for issues, Google Sheets QA workbook for cases / notes, and Xbox Game Bar + OBS for repro clips.
  • Deliverables: QA workbook (Sheets + PDF export), evidence clips on YouTube, and a Jira board with four verified tickets and thumbnails.
  • Result: Overall build felt stable but surfaced 4 high-impact input / hand-off defects, all 100% reproducible with clear steps and video.
  • Why this game? A fast, recognisable brawler with drop-in local co-op, perfect for testing input ownership and pause / join-in behaviour.

2: Charter-based Exploratory & Edge-Case Testing - Rebel Racing (Mobile)

    One-week exploratory / edge-case pass on a live F2P mobile racer, scoped to a single Android device.

    • Goal: Practise charter-based mobile QA with daily golden-path smoke tests plus targeted edge-case sessions.
    • Focus: Interruptions & recovery, LTE vs Wi-Fi, UI scaling/readability, input responsiveness, performance & device feel, and basic network / live surfaces (Store & Events).
    • Interesting result: Found a post-race rewards soft lock after an OS alarm + app close, plus smaller risks around LTE load delays and HUD readability.
    • Evidence: Moto g54 recordings (scrcpy/OBS) and a Google Sheets workbook with charters, bug log and STAR summary.

3: Regression Testing - Sworn (PC)

  • Goal: Run a focused one-week solo regression pass on a recent Sworn patch, confirming key fixes hold and catching any side-effects before the next update.
  • Scope: Patch-note changes and community-reported issues across save/load safety, quest and stamina systems, session start/quit flows, and UI readability in core hubs and menus.
  • Focus areas:
    • Change-driven charters mapped directly to specific patch-note items and known bugs.
    • Risk-baseline checks that always run: save/load integrity, session start/quit, and core progression.
    • Stamina, quests and rewards behaving correctly after fixes and across longer sessions.
    • UI readability and feedback around key states (low health, stamina, quest updates, errors).
    • Side-effect sweeps where fixes touch shared systems (combat, inventory, hub interactions).
  • Tools: Jira regression board, Excel/Sheets verification log, OBS before/after clips, and public patch notes + community threads used as change oracles.
  • Deliverables: Regression log, Fixed / Not Fixed / Reopened summary, verification matrix tied to patch-note items, and short before/after videos for key fixes.
  • Status: In Progress - regression workbook, board and oracles defined; playthrough and verification passes scheduled as the next portfolio project.
  • Why this game? Sworn is a systems-heavy, frequently updated action RPG, ideal for practising disciplined, patch-driven regression instead of ticking through a giant static test case list.

5: Educational UX & Cognitive Load Testing - Educational XR Lab (Browser-Based 3D Game-Design Tool)

  • Goal: Assess how well a browser-based 3D creation tool supports young / non-gamer learners: clear instructions, manageable cognitive load, basic accessibility (text size, contrast, icon clarity), and how easily kids can use it to learn structured thinking and light testing.
  • Scope: First-time onboarding, UI and goal visibility, learning flow and cognitive load, feedback and error recovery, and basic accessibility heuristics (text, contrast, language level, hit-targets).
  • Focus areas:
    • Onboarding & instructions: first-time clarity for non-gamers, ‚Äúwhat do I do now?‚Äù guidance, goal and next-step visibility.
    • Learning flow & cognitive load: pacing and ordering of concepts (sequence, repetition, selection, variables) and where learners stall, rush, or feel overwhelmed.
    • Feedback & error recovery: behaviour when blocks are missing, mis-connected, or mis-configured, and whether messaging feels encouraging and clear vs punishing or confusing.
    • Accessibility & readability: text size and contrast, icon clarity, language level, mouse/trackpad targeting, and usability on low-cost / low-spec devices.
  • Tools: Session charters built around learner journeys (‚Äúmake a character speak‚Äù, ‚Äútrigger an animation on click‚Äù), a heuristic checklist for onboarding + basic accessibility (lightly inspired by XAUR ideas), observation notes from a non-gamer / student test subject, and screen capture.
  • Deliverables: First-time learner journey map, an accessibility / UX findings document with practical recommendations, and an issue list grouped by onboarding, learning clarity, accessibility, and editor usability.
  • Status: Planned ‚Äî goals and focus areas defined; execution scheduled as a dedicated educational UX and cognitive load case study.
  • Why this game? Demonstrates applying games QA practices to a browser-based 3D game-design tool for young users, focusing on onboarding, learning flow, and accessibility for novice creators rather than only on core players.
  • Note: The specific product name is omitted by agreement; findings are shared privately with the development team, and a named public case study will only be published if they‚Äôre happy with it.

6: Multi-Input / Controller Parity QA - Recompile (PC)

  • Goal: Verify controller responsiveness, input-mapping clarity, and parity across two Xbox-compatible controllers on Recompile (PC Game Pass).
  • Scope: Mapping accuracy, input latency, haptics/rumble behaviour, Xbox prompt/icon alignment, hot-swap and multi-controller handling, and basic rebinding edge cases.
  • Focus areas:
    • Latency timing with video / stopwatch overlay in menus vs active gameplay.
    • Dead zones, drift, and axis ranges on each controller.
    • Button / axis parity plus correct on-screen Xbox icons and prompts.
    • Hot-swap ownership, join / leave flows, and any confusing error prompts.
    • Rebind conflicts, persistence, and reset-to-default behaviour.
    • Mode changes: input consistency between inventory/menus and live gameplay.
  • Tools: Google Sheets test plan, 2√ó Diswoe X360 controllers, OBS capture with stopwatch overlay.
  • Deliverables: Input-response matrix, a short latency report (ms table), a prompt/binding screenshot pack, and a STAR-style summary of the main findings.
  • Status: Planned ‚Äì test plan and tools defined; execution scheduled as a dedicated controller parity case study.
  • Why this game? Fast, precise movement and frequent mode switches make Recompile a good stress test for input mismatches, latency spikes, and inconsistent prompts.

7: Narrative / Localisation QA - Oxenfree (Mobile)

  • Goal: Evaluate subtitle timing, readability, tone consistency and key narrative choices in a branching mobile playthrough, with issues logged in an LQA-friendly format.
  • Scope: English source-language subtitles and UI text on Android (Netflix Games build). Audio quality, VO performance and non-English localisations are out of scope for this pass.
  • Focus areas: Subtitle timing and pacing, line breaks and overflow, speaker tags and attribution, tone and sensitive phrasing, and UI legibility on a mobile screen.
  • Detailed checks:
    • Subtitle pacing vs VO during walk-and-talk, radio tuning and interruptible dialogue.
    • Line breaks, truncation and scrolling captions in busy or fast-paced scenes.
    • Clear speaker attribution for overlapping lines and off-screen dialogue.
    • Empathy and inclusion in key choices and outcomes, aiming for reflection rather than pure confrontation.
    • Font size, scaling and contrast in dark scenes, glitch effects and UI overlays.
  • Tools: Google Sheets timing / localisation log, on-device Android recordings for VO/subtitle sync and branch coverage, mirrored capture (OBS / screen recorder) for evidence clips, and Android / OS accessibility tools for font scaling and contrast checks.
  • Deliverables: Subtitle and tone QA report, an accessibility checklist (readability, scaling, contrast), a localisation consistency log (terms, names, recurring phrases), and an LQA-style bug set with full lines, context and annotated screenshots or short clips.
  • Status: Planned ‚Äì narrative / localisation pass scheduled after the current mobile QA case study.
  • Why this game? Oxenfree‚Äôs walk-and-talk dialogue, radio tuning and branching supernatural story make timing, tone and readability critical, and it‚Äôs ideal for practising narrative and localisation QA on mobile.
